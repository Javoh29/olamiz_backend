# Olamiz — реестр решений (D1–D21)

Формат: решение → почему → что отвергнуто. Пересматривать можно, но не молча:
любое изменение — через обсуждение с владельцем проекта и правку этого файла.

---

**D1. Сегмент MVP — B2C, мелкий/средний инструмент.**
Крупная спецтехника (краны, экскаваторы) — другой бизнес (B2B, RFQ, договоры, страховки),
её тащить в MVP — размыть фокус. Отвергнуто: «всё от молотка до крана» в одном релизе.

**D2. Supply — managed, не открытая регистрация.**
Чистый P2P «любой выкладывает свою дрель» — статистическое кладбище (120+ закрытых платформ:
SnapGoods, NeighborGoods и др.; даже Fat Llama продался, не подняв раунд). Работает на
похожих рынках (Казахстан/Индия/ЮВА) агрегация существующих прокатчиков. Прокатчиков ищем
и подключаем сами, ручной онбординг. Частники не запрещены — проходят тот же фильтр.

**D3. География: пилот в одном районе Ташкента → город → области.**
Аренда гиперлокальна, ликвидность строится в каждой локации отдельно. 12 параллельных
холодных стартов силами соло-команды — самоубийство. Мультирегиональность в данных —
с 1-го дня, ограничение чисто операционное. Кандидаты района: Юнусабад/Сергели/Чиланзар.

**D4. Комиссия на старте ~0.**
Сначала ликвидность и привычка «сделка = кнопка», монетизация потом. При нулевой комиссии
у сторон нет мотива обходить платформу — окно для формирования привычки.

**D5. Своей логистики нет.**
Omni сжёг $40M на логистике. «Доставка» — атрибут прокатчика (есть/нет), фильтр в каталоге.

**D6. Сделка живёт в платформе; контакты — после подтверждения брони.**
Иначе строим «OLX с красивым каталогом»: без записи о сделке невозможны доказуемые отзывы,
рейтинг клиента, метрики, будущая монетизация. Телефон прокатчика скрыт до подтверждения;
клиентский телефон прокатчику — тоже только после подтверждения (анти-обход).

**D7. Деньги — офлайн (наличка/перевод между сторонами), платформа не участвует.**
Рынок наличный, эскроу на старте — трение и разработка без запроса. Эскроу/холд залога —
фаза N, архитектурно не блокируем.

**D8. Клиенты: app (Flutter) + SSR-витрина; прокатчик — только Telegram-бот; команда — SQLAdmin.**
Прокатчик живёт в Telegram — отдельное приложение ему не нужно и не взлетит. Админку не
пилить руками — SQLAdmin поверх готовых моделей.

**D9. Метрика скорости ответа прокатчика.**
Медиана по последним ~20 заявкам, до первого действия; отображение интервалами; новичкам —
бейдж «новый». Публична с MVP (дисциплинирует лучше звонков), фильтр по ней — фаза 2.
Отвергнуто: среднее (убивается выбросами), точные цифры (ложная точность).

**D10. Эскалация уведомлений: TG-пуш → SMS (~3 мин) → автозвонок (~7 мин, v1.1).**
Скорость подтверждения — главный операционный риск модели (рынок привык «позвонил — ответили»).
90% должно закрываться бесплатным TG-пушем; телефония — кнут. Параллельно клиенту — альтернативы.
Скорость ответа влияет на ранжирование.

**D11. Карточка = модель у прокатчика + количество; бронь из пула.**
Отвергнуто: карточка на каждую физическую единицу — прокатчик оперирует пулом («выдам ту,
что у двери»), клиенту нечем выбирать между №2 и №3, каталог замусоривается клонами.
Проблема «на фото красивый, привезли убитый» решается НЕ моделью каталога, а чек-листами,
правом отказа при выдаче и отзывом «соответствует фото». Под капотом сущность unit —
позже опциональный инвентарный учёт (приманка для серьёзных прокатчиков).

**D12. Регистрация клиента: телефон + SMS, без паспорта.**
Паспорт на уровне платформы — трение, убивающее конверсию; паспортные ритуалы остаются
в офлайне при выдаче (рынок так работает). Паспорт/гос-ID — поздние фазы.
Прокатчик: всё фиксируется на ручном онбординге (соглашение, паспорт/ИНН, реквизиты).

**D13. Чек-листы выдачи/возврата — обязательные, кнопки «пропустить» нет.**
Решение владельца: это фича безопасности, половинчатая обязательность её девальвирует;
загрузка точек невысокая, минута на фото — приемлемая цена. Страховочный болт: админ может
вручную закрыть зависшую сделку (форс-мажоры), обязательно с логом.

**D14. Споры: статус `dispute` → ручной разбор админкой по фото и чек-листам.**
Для MVP достаточно ручного процесса с доказательной базой. Автоматизация — по мере роста.

**D15. Публичная оферта: платформа — информационный сервис, договор аренды — между сторонами.**
Иначе первый спор о сломанном генераторе прилетает платформе. Стороны признают
фото/чек-листы/таймстампы фиксацией фактов. Пишет местный юрист под право РУз.
Акцепты: клиент — при регистрации, прокатчик — на онбординге. Версии оферты храним.

**D16. ПД граждан УЗ — на серверах внутри страны.**
Требование законодательства РУз о персональных данных. Влияет на выбор VPS и хранилища фото.
Внешним сервисам ПД не отдаём.

**D17. Отзывы двойные и привязаны к завершённой сделке.**
Клиент → прокатчик И конкретная карточка («соответствует фото: да/нет»); прокатчик → клиент.
Привязка к сделке = доказуемость (главное отличие от OLX, где отзывов нет вообще).
Публикация слепая (обе сдали или 7 дней) — защита от мести.

**D18. Стек: FastAPI + PostgreSQL + aiogram 3 + Redis/ARQ + Jinja2 SSR + SQLAdmin + Flutter.**
Один язык на всём бэкенде, общий domain для API/бота/витрины; модульный монолит в
Docker Compose на VPS. Отвергнуто: MongoDB (календарь броней — реляционная задача),
Celery (гаубица по воробьям), Kubernetes/микросервисы (нет нагрузки), ручная админка
(пожиратель месяца).

**D19. SEO-витрина — только серверный HTML; Flutter Web в MVP отсутствует.**
Flutter Web рендерит в canvas — Google не индексирует контент, Core Web Vitals страдают;
официальная позиция Flutter: для контентных сайтов — HTML. Органика из Google — единственный
бесплатный канал спроса при нулевом маркетинговом бюджете, убивать её нельзя.
Витрина = 3–4 Jinja-шаблона поверх готовых API. «Забронировать» → deep link в app.
Flutter Web (адаптация мобильного кода) — фаза 2+, поддомен app.olamiz.uz, без SEO-обязанностей.

**D20. Пилот: район из тройки Юнусабад/Сергели/Чиланзар; успех = fulfillment >50% и ≥100 сделок/мес.**
Пороги — к 2–3 месяцу пилота, не к первому. Недобор — сигнал менять район/категории/онбординг.

**D21. Названия: Olamiz (клиентский бренд) + Beramiz (бренд прокатчиков).**
Одна платформа, два фасада: **Olamiz** (olamiz.uz, узб. «берём») — app и SEO-витрина
для клиентов; **Beramiz** (beramiz.uz, узб. «даём») — Telegram-бот и лендинг онбординга
для прокатчиков. Оба домена забронированы. Пара осмысленна для узбекоязычной аудитории,
оба слова — три слога, диктуются по телефону однозначно, латиница без разночтений,
не привязаны к инструментам (масштабируются на спецтехнику).
Сделать до печати оферты: проверить товарные знаки «Olamiz» и «Beramiz» в базе
Агентства ИС РУз; занять @olamiz_uz и @beramiz_bot в Telegram; в сторах публиковаться
как «Olamiz — аренда инструментов». Заменяет прежнее рабочее название проекта.
