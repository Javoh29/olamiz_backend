# Olamiz — продуктовая спецификация (MVP)

Версия: 1.1 · Июль 2026 · Статус: утверждена для разработки

## 1. Продукт одним абзацем

Одна платформа аренды строительного инструмента в Узбекистане, два бренда-фасада:
**Olamiz** (olamiz.uz, узб. «берём») — для клиентов (mobile app + SEO-витрина) и
**Beramiz** (beramiz.uz, узб. «даём») — для прокатчиков (Telegram-бот + лендинг
онбординга «сдавайте инструмент через Olamiz»). Оба домена забронированы.
Модель — managed-агрегатор: прокатчиков (юрлица и опытные частники) платформа находит и
подключает сама через ручной онбординг. Клиент находит инструмент в каталоге, бронирует,
прокатчик подтверждает — после этого стороны получают контакты друг друга и завершают сделку
офлайн (оплата и залог наличными, самовывоз или доставка силами прокатчика). Платформа
фиксирует жизненный цикл сделки, состояние инструмента (фото + чек-листы) и взаимные отзывы.
Комиссия на MVP — 0; монетизация — после набора ликвидности.

**Чем НЕ является:** платёжным сервисом (в деньги не лезем), логистикой (не доставляем),
открытой доской объявлений (self-service регистрации прокатчиков нет).

## 2. Рынок и география

- Пилот: **один район Ташкента** (кандидаты: Юнусабад, Сергели, Чиланзар — выбрать по
  плотности supply при онбординге). Затем: город целиком → области.
- Мультирегиональность (регион → город → район) заложена в данные и фильтры с 1-го дня,
  ограничение пилота — операционное, не техническое.
- Сегмент MVP: B2C, мелкий/средний инструмент (перфораторы, болгарки, генераторы,
  виброплиты и т.п.). Крупная спецтехника (краны, экскаваторы, B2B/RFQ) — отдельная фаза N.

**Метрики успеха пилота** (горизонт 2–3 месяца после запуска):
- fulfillment rate > 50% (доля созданных заявок, дошедших до выдачи);
- ≥ 100 подтверждённых сделок/мес.
Недобор — сигнал менять район/категории/онбординг, а не хоронить проект.

## 3. Роли и клиенты платформы

| Роль | Интерфейс | Что делает |
|---|---|---|
| Клиент (арендатор) | **Flutter mobile app** (iOS/Android) | Поиск, каталог, бронь, чек-листы, отзывы, история |
| Клиент (аноним из поиска) | **SSR-витрина olamiz.uz** | Смотрит каталог/карточки из Google, «Забронировать» → deep link в app |
| Прокатчик | **Только Telegram-бот (бренд Beramiz)** | Заявки (подтвердить/отклонить), календарь занятости, управление позициями, чек-листы выдачи/возврата |
| Команда платформы | **Админка (SQLAdmin)** | Онбординг прокатчиков, модерация карточек, споры, принудительное закрытие сделок, метрики |

Веб-приложение для клиента (Flutter Web, поддомен app.olamiz.uz) — фаза 2+, без SEO-обязанностей.

## 4. Регистрация и идентификация

**Клиент:** телефон + SMS-код (OTP). Имя — опционально. Паспорт на уровне платформы НЕ
требуется (паспортные дела — офлайн между сторонами при выдаче, как рынок привык).
При регистрации — акцепт публичной оферты (галочка, версия оферты фиксируется).

**Прокатчик:** только ручной онбординг командой платформы. Фиксируем: название точки,
телефон(ы), локация (район, адрес), тип (юрлицо/частник), ИНН или паспортные данные,
реквизиты (на будущее), признак «доставка есть/нет». Подписывает соглашение о работе
с платформой (акцепт оферты прокатчика). Привязка Telegram-аккаунта к боту — по
одноразовому коду, выданному при онбординге.

## 5. Каталог

- **Дерево категорий создаёт и контролирует платформа.** 2 уровня, максимум 3.
  Прокатчик не может создавать категории. Названия — ru/uz.
- **Карточка = модель инструмента у конкретного прокатчика** («Перфоратор Bosch GBH 2-26»
  у прокатчика X) с полем **«количество»**. Бронь резервирует единицу из пула.
- Поля карточки: категория, название/модель, описание, фото, цена/сутки, залог (сумма или
  «паспорт»/«без залога»), количество, район (наследуется от точки прокатчика).
- Атрибуты категорий (мощность, патрон и т.п.) и фильтры по ним — фаза 2. В MVP —
  категория + текстовый поиск.
- **Модерация карточек** — на онбординге и при изменениях: платформа причёсывает названия,
  вырезает телефоны из фото/описаний. Карточка публикуется после одобрения.
- Под капотом — сущность `unit` (физическая единица), создаётся автоматически, в MVP
  никому не видна. Позже — опциональный инвентарный учёт для прокатчиков (фаза 2+).

**Витрина (SEO):** страницы категории / карточки / прокатчика — серверный HTML со своими URL,
title/description, OpenGraph (превью в Telegram). Кнопка «Забронировать» → deep link в app
(установлено — открывает карточку в app; нет — стор + deferred deep link).

## 6. Сделка — основной флоу

Сделка (booking) — центральная сущность платформы. Телефон прокатчика скрыт до подтверждения.

1. **Заявка.** Клиент в app: карточка → даты → комментарий → «Забронировать».
   Заявка НЕ блокирует остаток (защита от зависших заявок).
2. **Уведомление прокатчика** с эскалацией:
   Telegram-пуш (мгновенно) → SMS через ~3 мин → автозвонок через ~7 мин (звонок — v1.1).
   Если ответа нет ~15 мин — клиенту показываются альтернативы из каталога.
   Заявка без ответа истекает через 24 ч (expired).
3. **Подтверждение.** Прокатчик в боте: «Подтвердить» / «Отклонить».
   При подтверждении система атомарно проверяет доступность пула на даты
   (пересечения с confirmed/active бронями); если единиц не осталось — авто-отказ с пояснением.
   После подтверждения **обе стороны видят контакты друг друга** — договариваются о времени,
   деньгах, доставке офлайн.
4. **Выдача.** При передаче инструмента — обязательный чек-лист выдачи (см. §7).
   Сделка → `active`.
5. **Возврат.** Чек-лист возврата. Всё ок → `completed`; прокатчик пометил проблему → `dispute`.
   Если прокатчик не закрыл сделку через 3 дня после даты окончания — автозавершение
   с уведомлением обеих сторон.
6. **Отзывы.** После `completed` обе стороны получают запрос на отзыв (см. §8).

Отмена: клиент может отменить до выдачи; прокатчик может отменить подтверждённую бронь
(портит его метрики). Все отмены логируются с причиной.

## 7. Чек-листы выдачи и возврата (обязательные)

Цель: доказательная база состояния инструмента. Кнопки «пропустить» НЕТ.

**Выдача (обе стороны):**
- фото инструмента 2–4 шт (клиент и прокатчик, каждый со своей стороны);
- чек-лист 3–5 галочек: включается/работает штатно; комплектность (кейс, оснастка);
  внешних повреждений сверх фото нет; (галочки конфигурируются платформой, общие для всех
  категорий в MVP; категорийные — фаза 2).

**Возврат (зеркально):** прокатчик подтверждает «работает, комплект полный, новых
повреждений нет» либо помечает проблему (+фото). Проблема → статус `dispute`.

Клиент вправе отказаться от аренды на этапе выдачи без штрафа, если состояние заметно
хуже фото карточки (сделка → отменена с причиной «состояние не соответствует»).

Правило споров (в оферте): нет чек-листа выдачи — претензии по состоянию не рассматриваются.
Зависшие сделки (форс-мажор, недоступность сторон) вручную закрывает админ — с обязательным
логом «кто/когда/почему».

## 8. Отзывы и рейтинги

- Отзыв существует **только у завершённой сделки** — это главный анти-фейк механизм.
- Клиент оценивает: (а) прокатчика (звёзды + текст) и (б) конкретную карточку —
  «состояние соответствует фото: да/нет» + звёзды.
- Прокатчик оценивает клиента (звёзды + текст). Рейтинг клиента строится из истории сделок —
  прокатчик видит «12 аренд, проблем не было» и сам решает вопрос залога.
- Публикация — двойная слепая: отзывы открываются после сдачи обеими сторонами либо через
  7 дней (защита от отзывов-мести).
- Заврaвшаяся карточка (низкий % «соответствует фото») тонет в выдаче.

## 9. Метрика «скорость ответа прокатчика»

- Медиана по последним ~20 заявкам; считается время от создания заявки до ПЕРВОГО действия
  (подтвердил/отклонил).
- Отображение интервалами: «обычно отвечает за ~10–15 мин / в течение часа / в течение дня /
  отвечает долго». Точные цифры не показываем.
- < 5 заявок в истории → бейдж «новый на платформе» вместо метрики.
- Публична в карточке с MVP. Фильтр/сортировка по ней — фаза 2.
- Скорость ответа влияет на позицию в выдаче каталога (см. ранжирование в architecture.md).

## 10. Юридика

- **Публичная оферта:** платформа — информационный сервис; договор аренды заключается между
  клиентом и прокатчиком напрямую. Стороны признают фото/чек-листы/таймстампы платформы
  фиксацией фактов. Порядок споров: статус «спор» → разбор командой платформы по материалам
  сделки; платформа модерирует, но не суд.
- Акцепты: клиент — при регистрации; прокатчик — на онбординге (подписывает соглашение).
- Текст оферты пишет **местный юрист под законодательство РУз** (не шаблон из интернета).
- **ПД граждан УЗ хранятся на серверах внутри страны** — жёсткое требование к хостингу
  и хранилищу фото.

## 11. Скоуп по фазам

### v1.0 (MVP, первый релиз)
- Клиентский Flutter app: регистрация (телефон+OTP), каталог (категории, поиск, фильтр
  по району), карточка, бронирование, статусы сделки, чек-листы (фото+галочки), отзывы,
  история сделок. Локализация ru/uz.
- Telegram-бот прокатчика: привязка по коду, заявки (подтвердить/отклонить), список
  предстоящих выдач/возвратов (календарь списком), чек-листы выдачи/возврата,
  пауза/цена/количество по своим карточкам.
- SSR-витрина: главная, категории, карточки, страницы прокатчиков, sitemap, OG-теги,
  deep links.
- Эскалация уведомлений: Telegram-пуш → SMS. Показ альтернатив при молчании прокатчика.
- Метрика скорости ответа (публично в карточке), рейтинги, двойные отзывы.
- Админка: прокатчики, модерация карточек, сделки (+принудительное закрытие с логом),
  споры, версии оферты, базовые метрики (SQL-дашборд).
- Статус «спор» + ручной разбор.

### v1.1 (сразу после запуска, по живому фидбеку)
- Автозвонок в эскалации.
- Push-уведомления клиенту (FCM) о смене статусов.
- Улучшение поиска (опечатки, синонимы ru/uz).
- Календарь занятости прокатчика в виде календаря (не списка).
- Лендинг для прокатчиков на beramiz.uz («сдавайте через Olamiz» + заявка на онбординг).

### Фаза 2
- Фильтр/сортировка по скорости ответа; атрибуты категорий и фильтры по ним.
- Flutter Web (app.olamiz.uz) — веб-бронирование, адаптация мобильного кода.
- Опциональный инвентарный учёт по единицам (unit) для прокатчиков.
- Instant booking для проверенных прокатчиков.
- Расширение географии: Ташкент целиком → области (по плейбуку пилота).

### Фаза N (после ликвидности)
- Монетизация (комиссия), платежи/эскроу через Payme/Click/Uzum, залог холдом.
- B2B/RFQ-ветка для крупной спецтехники.
- Страхование дорогих позиций.

## 12. Открытые вопросы (решить до/во время разработки)

1. Конкретный район пилота — выбрать по результатам онбординга первых прокатчиков.
2. Хостинг-провайдер в УЗ (аптайм, S3-совместимое хранилище) — отдельное мини-исследование.
3. Юрист и сроки готовности оферты — до публичного запуска.
4. Проверка товарных знаков «Olamiz» и «Beramiz» в базе Агентства ИС РУз — до печати оферты; занять @olamiz_uz и @beramiz_bot в Telegram.
5. SMS-провайдер (Eskiz vs PlayMobile) — по ценам и API на момент интеграции.
6. Язык SSR-витрины в v1.0: только ru или ru+uz сразу (влияет на SEO-охват).
