# Olamiz — Backend: руководство разработки

Версия: 1.0 · Дополняет `architecture.md` (система в целом) деталями реализации бэкенда.

## 1. Зона ответственности

Одно FastAPI-приложение + ARQ worker. Четыре адаптера поверх общего domain-слоя:
REST API для mobile (`api/`), Telegram-бот прокатчика (`bot/`), SSR-витрина (`web/`),
админка (`admin/`). Вся бизнес-логика — в `domain/`, адаптеры тонкие.

## 2. Структура

```
backend/app/
├── core/           # config (pydantic-settings), db (async engine/session),
│                   # security (JWT, OTP), deps, storage (S3/MinIO), sms (SmsGateway)
├── domain/
│   ├── users/      # OTP-аутентификация, профиль, рейтинг клиента
│   ├── suppliers/  # прокатчики, привязка TG по коду, метрика ответа
│   ├── geo/        # регионы/города/районы (справочники)
│   ├── catalog/    # категории, listings, модерация, units, ranking.py
│   ├── booking/    # статусная машина, доступность, отмены — СЕРДЦЕ СИСТЕМЫ
│   ├── checklists/ # чек-листы + фото
│   ├── reviews/    # слепые отзывы, рейтинги
│   ├── notifications/  # эскалации, шаблоны ru/uz, лог
│   └── disputes/
├── api/v1/         # роутеры REST
├── bot/            # aiogram-роутеры + FSM чек-листов
├── web/            # SSR: роутеры + templates/ (Jinja2) + static/
├── admin/          # SQLAdmin views
└── worker.py       # ARQ: задачи и cron
```

## 3. Ключевые флоу (пошагово)

### 3.1 Создание заявки
1. Валидации: listing активен; date_start ≥ сегодня; date_end > date_start;
   разумный максимум срока (конфиг, напр. 30 дней); у клиента нет другой pending/confirmed
   заявки на этот же listing с пересекающимися датами (анти-дубль).
2. Мягкая проверка доступности (для UX): если пул уже занят — сразу честное «на эти даты занято».
   Остаток НЕ резервируется (резервируют только confirmed/active).
3. `Booking(status=pending)` → планирование цепочки эскалаций в ARQ → TG-пуш прокатчику
   с inline-кнопками.

### 3.2 Подтверждение (критичная транзакция)
```
async with session.begin():
    listing = SELECT ... FOR UPDATE          # лочим карточку
    booked  = COUNT confirmed/active броней с пересечением [s, e)
    if booked >= listing.quantity: -> авто-отказ "занято", уведомить клиента
    booking.confirm()                        # статусная машина, first_action_at
# после коммита: отмена оставшихся эскалаций, открытие контактов,
# SMS клиенту «Бронь подтверждена, прокатчик: имя, телефон» (одна SMS на сделку),
# пересчёт response_median_sec прокатчика (ARQ)
```
Отклонение: `booking.decline(reason?)`, эскалации отменяются, клиенту — уведомление
с подборкой альтернатив.

### 3.3 Выдача
Сделка `confirmed` → обе стороны сдают чек-лист выдачи (клиент — из app, прокатчик — из бота):
фото 2–4 + галочки. Когда сданы ОБА отчёта → `active` (issued_at). Кнопки «пропустить» нет;
застрявшую выдачу разруливает админ (force-действие с логом).
Отказ клиента на выдаче («состояние хуже фото») → `cancelled` с причиной, без чек-листов.

### 3.4 Возврат
Прокатчик в боте: «Принять возврат» → фото + галочки → всё ок → `completed`
(returned_at) → обеим сторонам запрос отзыва. Помечена проблема → `dispute` +
уведомление команде. Клиент может приложить свои фото возврата (опционально, в app).

### 3.5 Отзывы (слепая публикация)
После `completed` обе стороны могут сдать отзыв. Публикация: когда сдали обе ИЛИ через
7 дней (ARQ-задача). До публикации отзывы не видны никому, включая автора второй стороны.

## 4. ARQ worker — задачи

| Задача | Триггер | Действие |
|---|---|---|
| `escalate_sms` | +3 мин после заявки | если `first_action_at IS NULL` → SMS прокатчику |
| `escalate_call` | +7 мин (v1.1) | автозвонок |
| `offer_alternatives` | +15 мин | пуш клиенту в app с подборкой похожих |
| `expire_booking` | +24 ч | pending → expired, уведомления |
| `auto_complete` | date_end + 3 дня | active → completed (+уведомления, пометка auto) |
| `publish_reviews` | +7 дней после completed | публикация несданных пар отзывов |
| `recalc_supplier_metrics` | после first_action / cron | медиана ответа (последние 20), рейтинги |
| `review_reminder` | +24 ч после completed | напоминание об отзыве (по одному разу) |

Все интервалы — из конфига. Каждая отправка — строка в `notifications_log`.

## 5. API v1 — эндпоинты

Auth: `POST /auth/otp/request` (rate limit: 3/тел/час, 10/IP/час), `POST /auth/otp/verify`
→ JWT access (30 мин) + refresh (30 дней), `POST /auth/refresh`.

| Метод | Путь | Описание |
|---|---|---|
| GET | /catalog/categories | дерево, ru/uz |
| GET | /catalog/listings | фильтры: category, district, q, date_from/to; сортировка по score |
| GET | /catalog/listings/{id} | карточка; `supplier.phone` — ТОЛЬКО если у клиента есть подтверждённая бронь этой карточки |
| GET | /catalog/listings/{id}/availability | занятые интервалы для календаря дат |
| POST | /bookings | создать заявку |
| GET | /bookings/my | список с фильтром по статусу |
| GET | /bookings/{id} | деталка; контакты — по статусу |
| POST | /bookings/{id}/cancel | отмена клиентом (до active) |
| POST | /bookings/{id}/checklist | multipart: kind=issue, фото + галочки |
| POST | /bookings/{id}/review | звёзды прокатчику, звёзды+matches_photo карточке, текст |
| GET/PATCH | /me | профиль, язык |

Скрытие телефона реализуется в domain/Pydantic-схеме (поле условное), НЕ на клиенте.
Фото — multipart через API → MinIO (presigned-выдача на чтение). Ошибки — единый формат
`{code, message_ru, message_uz}`.

## 6. Telegram-бот прокатчика (aiogram 3, бренд Beramiz)

Бот живёт под брендом Beramiz (@beramiz_bot — проверить/занять). Все тексты бота —
от имени Beramiz, упоминая Olamiz как клиентскую витрину («клиент нашёл вас в Olamiz»).

- **Привязка:** `/start <code>` — одноразовый код с онбординга → supplier.tg_chat_id.
- **Заявка:** карточка заявки (что, даты, район клиента, комментарий, рейтинг клиента
  и число его сделок) + inline «✅ Подтвердить / ❌ Отклонить». После подтверждения —
  контакты клиента.
- **Чек-листы:** FSM-сценарий: «пришлите 2–4 фото» → приём photo → галочки
  inline-клавиатурой → подтверждение. Таймаут FSM — 30 мин, потом можно начать заново.
- **Команды:** `/today` (выдать/принять сегодня), `/bookings` (ближайшие брони),
  `/listings` (свои карточки: пауза/цена/количество — изменение цены и количества
  применяется сразу, правка текста/фото уходит на модерацию).
- Язык бота — ru/uz по выбору прокатчика при привязке.

## 7. SSR-витрина (SEO) — olamiz.uz

Маршруты: `/`, `/c/{category_slug}`, `/l/{listing_slug}`, `/s/{supplier_slug}`,
`/sitemap.xml`, `/robots.txt`.

Домены: nginx обслуживает оба — **olamiz.uz** (витрина) и **beramiz.uz** (v1.0 —
редирект на страницу «стать прокатчиком» витрины или заглушка; v1.1 — полноценный
лендинг онбординга прокатчиков с формой заявки).

Требования:
- title/description по шаблонам: «Аренда {название} в {район} — от {цена} сум/сутки | Olamiz».
- OG-теги на карточках (фото, цена) — превью в Telegram.
- «Забронировать» → `https://olamiz.uz/l/{slug}?open=app` — App Links/Universal Links;
  без приложения → интерстициал со ссылками на сторы.
- Телефоны прокатчиков на витрине НЕ отображаются нигде.
- Никакого JS-фреймворка; суммарно CSS+JS < 50 КБ; server-side пагинация.
- В `sitemap.xml` — только активные listings и категории с товаром.

## 8. Админка (SQLAdmin)

Views: Suppliers (создание = онбординг, генерация кода привязки), Listings
(модерация: draft/moderation → active, правки), Bookings (просмотр, force-close
с обязательным комментарием — пишется в admin_log), Disputes (материалы сделки:
фото/чек-листы, резолюция), Users, Categories, Districts, OfferVersions, Settings
(тайминги эскалаций, веса ранжирования).

## 9. Конфиг (env / settings)

`ESCALATE_SMS_MIN=3`, `ESCALATE_CALL_MIN=7`, `ALTERNATIVES_MIN=15`,
`BOOKING_EXPIRE_H=24`, `AUTO_COMPLETE_DAYS=3`, `REVIEW_BLIND_DAYS=7`,
`MAX_RENT_DAYS=30`, `OTP_TTL_MIN=5`, веса ранжирования `RANK_W1..W4`,
ключи SMS-провайдера, S3, Sentry, TG-токен.

## 10. Тестирование (обязательный минимум)

1. Статусная машина: все разрешённые/запрещённые переходы.
2. Доступность: гонка двух подтверждений на последнюю единицу (два параллельных
   confirm — один обязан получить авто-отказ).
3. Скрытие телефона: до/после подтверждения, на витрине.
4. Эскалации: отмена цепочки после first_action.
5. Слепые отзывы: невидимость до публикации, публикация по обоим/по таймеру.
6. OTP: rate limits, TTL.

## 11. Definition of Done (фича бэкенда)

Код в domain + тонкий адаптер · Alembic-миграция · тесты на логику · тексты ru/uz
в словарях · ошибки в едином формате · обновлён relevant doc, если поведение изменилось.
