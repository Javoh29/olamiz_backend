# Olamiz — Mobile (Flutter): руководство разработки

Версия: 1.0 · Приложение клиента-арендатора (uz.olamiz.app). Все транзакции — только здесь.
Прокатчик приложения НЕ имеет (у него Telegram-бот).

## 1. Стек и правила

- Flutter stable, Dart 3, null safety строго.
- **State:** Riverpod (провайдеры по фичам, без глобального god-state).
- **Навигация:** go_router + deep links (`https://olamiz.uz/l/{slug}`, `olamiz://booking/{id}`).
- **Сеть:** dio + json_serializable; интерцепторы: JWT + авто-refresh, единый маппинг ошибок.
- **Локализация:** ru/uz через ARB с первого экрана. Строк в коде нет.
- Тема — light only (MVP), Material 3, токены из `docs/design.md`.

## 2. Структура

```
lib/
├── app/                  # MaterialApp, router, theme, l10n
├── core/                 # api-клиент, storage (secure), ошибки, утилиты, виджеты-базы
└── features/
    ├── auth/             # телефон → OTP → JWT
    ├── catalog/          # категории, список, поиск, фильтр района
    ├── listing/          # карточка, календарь доступности
    ├── booking/          # создание брони, мои сделки, деталка, отмена
    ├── checklist/        # чек-лист выдачи (фото + галочки)
    ├── reviews/          # отзыв после сделки
    └── profile/          # профиль, язык, история, оферта
# каждая фича: data/ (dto, repo) · domain/ (модели, логика) · presentation/ (экраны, провайдеры)
```

## 3. Карта экранов

```
Splash → (нет токена) Onboarding (2–3 слайда) → Auth(телефон → OTP)
      → (есть токен) Home

Home (bottom nav): Каталог · Мои сделки · Профиль

Каталог → Категория → Список карточек → Карточка → Выбор дат → Подтверждение заявки
Мои сделки → Деталка сделки → [по статусу]: Отменить / Чек-лист выдачи / Отзыв
```

Правила:
- Каталог смотрится БЕЗ авторизации; логин требуем только на «Забронировать»
  (после логина возвращаем на ту же карточку — не терять контекст).
- Фильтр района — заметный, вверху каталога; выбор запоминается.

## 4. Ключевые экраны — требования

**Карточка:** галерея фото, цена/сутки крупно, залог, бейдж «доставка», район,
блок прокатчика (имя, рейтинг, «обычно отвечает …» ИЛИ бейдж «новый на платформе»),
отзывы (с «соответствует фото: N%»), календарь занятых дат. Кнопка «Забронировать» —
sticky снизу. Телефона прокатчика на карточке НЕТ — не рисовать даже placeholder.

**Создание брони:** range-выбор дат (занятые — задизейблены, данные из
`/availability`), итого = цена × дней (справочно, «оплата прокатчику напрямую»),
комментарий. После отправки — экран статуса «Ждём подтверждения» с честным
объяснением и таймером; при молчании прокатчика (пуш `alternatives`) — блок
«Похожие варианты».

**Деталка сделки:** статус-таймлайн (заявка → подтверждена → выдан → завершена),
действия строго по статусу. После подтверждения — карточка контакта прокатчика
(имя, телефон, кнопки позвонить/в Telegram). При `dispute` — статус «Разбирается
командой Olamiz» без действий.

**Чек-лист выдачи (критичный UX):** пошагово: (1) камера сразу, 2–4 фото,
превью, пересъёмка; (2) галочки крупным текстом (включается и работает /
комплектность / повреждений сверх фото нет); (3) отправка. Пропустить нельзя —
кнопки нет. Отдельная понятная кнопка «Отказаться от аренды» (состояние хуже фото) —
с подтверждением и выбором причины.

**Отзыв:** звёзды прокатчику · звёзды карточке + переключатель «соответствует фото:
да/нет» · текст (опционально). Один экран, не мастер.

## 5. Статусы и обновления

- MVP: pull-to-refresh + авто-поллинг деталки активной сделки (каждые 30 сек на экране).
  О подтверждении клиент дополнительно узнаёт по SMS (шлёт бэкенд).
- v1.1: FCM-пуши о смене статусов (в payload — только booking_id, без ПД); поллинг остаётся фолбэком.

## 6. Фото чек-листов

- Съёмка только с камеры (не галерея) — фото фиксирует момент передачи.
- Сжатие на клиенте: ~1280px по длинной стороне, JPEG q80 (flutter_image_compress).
- Загрузка multipart в API с ретраями; при обрыве — фото в локальной очереди,
  повтор при следующем открытии (сделка не должна зависнуть из-за 3G).

## 7. Ошибки, лоадеры, пустые состояния

- Единый маппинг ошибок API → человеческий текст ru/uz; коды не показывать.
- Skeleton-лоадеры в каталоге; на действиях — блокировка кнопки + прогресс.
- Пустые состояния с действием: «В этом районе пока пусто — посмотреть весь Ташкент».
- Оффлайн: баннер «Нет соединения», кэш последнего каталога — nice to have, не блокер.

## 8. Definition of Done (фича мобилки)

Feature-структура соблюдена · строки в ARB (ru+uz) · состояния loading/error/empty
покрыты · deep link на экран работает (если применимо) · тесты на domain-логику фичи ·
проверено на дешёвом Android (низкая память, медленная сеть).
