# Olamiz — контекст проекта

Маркетплейс аренды строительного инструмента в Узбекистане. Одна платформа, два бренда:
- **Olamiz** (olamiz.uz, узб. «берём») — клиентская сторона: mobile app, SEO-витрина.
- **Beramiz** (beramiz.uz, узб. «даём») — сторона прокатчиков: Telegram-бот, лендинг онбординга.

Связывает клиентов и прокатчиков: каталог → бронь → подтверждение → офлайн-сделка → отзывы.
Платформа НЕ участвует в деньгах (оплата/залог — наличкой между сторонами) и НЕ доставляет.
Ценность платформы: каталог с категориями, бронирование, доказуемые отзывы, фото-чек-листы, рейтинги.

## Ключевые принципы (нарушать нельзя)

1. **Сделка живёт в платформе.** Телефон прокатчика скрыт до подтверждения брони. Контакты
   открываются только после подтверждения. Отзыв существует только у завершённой сделки.
2. **Фото-чек-листы выдачи/возврата обязательны.** Кнопки «пропустить» нет. Закрыть зависшую
   сделку может только админ (с логом причины).
3. **Комиссия 0 на MVP.** Не проектировать платежи/эскроу сейчас, но не блокировать их появление.
4. **ПД граждан УЗ хранятся на серверах внутри Узбекистана.** Никаких внешних SaaS для хранения
   персональных данных и фото сделок.
5. **SEO-витрина — только серверный HTML (Jinja2).** Никакого Flutter Web / JS-рендеринга
   на публичном каталоге.
6. **Мультирегиональность в данных с 1-го дня** (регион → город → район), даже если пилот — один район Ташкента.

## Стек

- **Backend:** Python 3.12+, FastAPI, SQLAlchemy 2.0 (async), Alembic, Pydantic v2
- **БД:** PostgreSQL 16; **очереди/отложенные задачи:** Redis + ARQ
- **Telegram-бот прокатчика:** aiogram 3, общая кодовая база с API (общий domain-слой)
- **SSR-витрина (SEO):** Jinja2 из того же FastAPI-приложения
- **Админка:** SQLAdmin поверх тех же моделей
- **Mobile:** Flutter (Dart 3), feature-based структура, Riverpod; все транзакции только в app
- **SMS:** Eskiz / PlayMobile (провайдер конфигурируется)
- **Деплой:** Docker Compose на VPS в Узбекистане, nginx, S3-совместимое хранилище фото (MinIO)

## Конвенции

- **Python:** ruff (lint+format), mypy strict на domain-слое, pytest (+pytest-asyncio).
  Бизнес-логика — в `domain/`, тонкие хендлеры в `api/`/`bot/`/`web/`. Никакой логики в роутерах.
- **Статусы сделки — только через статусную машину** (`domain/booking`), прямые UPDATE статуса запрещены.
- **Flutter:** feature-based (`lib/features/<feature>/{data,domain,presentation}`), Riverpod,
  go_router, null safety строго. Локализация ru/uz через ARB с 1-го дня.
- **Миграции:** любое изменение моделей = Alembic-миграция в том же PR.
- **Тексты пользователю** (бот, SMS, app) — не хардкодить, держать в словарях локализации ru/uz.
- Время храним в UTC; таймзона отображения — Asia/Tashkent.
